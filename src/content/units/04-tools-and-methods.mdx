This unit is about the **specific tools and techniques** attackers use. For each one, a good answer explains **what it is, how it works, why criminals use it, an example, and how to defend against it.**

## Proxy servers

**A proxy server is an intermediary system that sits between a user's device and the internet, forwarding requests on the user's behalf.** Used legitimately it provides privacy, caching, and content filtering; abused by criminals it is a tool to **hide their real IP address and physical location**.

### How attackers use it

When a criminal routes an attack through a proxy, the target's logs record the **proxy's IP address**, not the attacker's real one. To make tracing almost impossible, attackers **chain** several proxies across different countries, so each hop hides the previous one — law enforcement would have to unravel every link, often across legal jurisdictions.

<ProxyChain />

### Purpose in cybercrime

- **Anonymity** — concealing identity and location during an attack.
- **Bypassing restrictions** — slipping past firewalls, IP whitelists, or geo-blocks.

<Callout type="tip">
**Example:** a hacker in Russia routes through proxies in Brazil and Germany to attack a UK firm, making it look as if the attack came from South America. A proxy is **not illegal in itself** — it becomes suspicious only when used to conceal crime.
</Callout>

## Password checking (cracking / guessing)

**Password checking, technically called password cracking, is the use of automated methods to systematically guess or recover user credentials** in order to gain unauthorized access to an account or system.

### Main methods

- **Brute-force attack** — automated software tries *every possible combination* of letters, numbers, and symbols until it finds the password. Guaranteed to work eventually, but slow against long passwords.
- **Dictionary attack** — instead of every combination, the tool tries a prepared list of common words, names, and leaked passwords (`password123`, `qwerty`). Much faster because most people choose weak passwords.
- **Credential stuffing** — taking username/password pairs **stolen in one breach** and automatically trying them on many other websites, exploiting the habit of **password reuse**.
- **Rainbow-table attack** — using precomputed tables of hashes to reverse stolen password hashes quickly.

### Why it works and how to defend

It succeeds because people pick weak, reused passwords. Defences: **strong, unique passwords**, **multi-factor authentication (MFA)**, **account lockout** after failed attempts, **salting and hashing** of stored passwords, and rate-limiting login attempts.

## Random checking (vulnerability scanning / probing)

**Random checking is an automated reconnaissance technique in which attackers sweep across large ranges of random IP addresses or networks, looking for unpatched software, open ports, or weak configurations.** Unlike a targeted attack, it is **opportunistic** — the attacker does not care *who* the victim is, only that a weakness exists.

### How it works

Automated scanners broadcast requests to thousands of random servers and wait to see which ones respond with an exploitable weakness. Any vulnerable machine that replies is flagged for a follow-up exploit. This is the opening phase of mass attacks such as worm and **ransomware** outbreaks.

<Callout type="tip">
**Example:** a botnet scans thousands of web servers for an outdated, vulnerable version of Apache; every unpatched server that responds is queued for an automatic exploit.
</Callout>

## Trojan horses and backdoors

- **A Trojan horse is malicious software disguised as something legitimate and useful** (a game, a free PDF reader, a utility). It does **not** self-replicate like a virus — it depends on the user being tricked into running it.
- **A backdoor is a hidden method of entering a system that bypasses normal authentication**, letting an attacker get in secretly and repeatedly.

### How they work together

Attackers often **hide a backdoor inside a Trojan**. When the victim installs the "useful" software, the Trojan executes and quietly opens the backdoor. From then on the attacker can log in remotely at any time to upload more malware, log keystrokes, or steal files — **without the user's knowledge**, and even after the original vulnerability is patched.

<Callout type="tip">
**Example:** a free "utility" really does its advertised job, but secretly opens network **Port 4444** (a backdoor), giving an off-site attacker full remote control of the desktop. Memory hook: a Trojan is a *gift that hides danger inside*.
</Callout>

## DoS and DDoS attacks

- **A Denial-of-Service (DoS) attack tries to make a machine, website, or network unavailable to its legitimate users** by overwhelming it with traffic or requests so it slows down or crashes.
- **A Distributed Denial-of-Service (DDoS) attack achieves the same goal but launches from many compromised devices at once** — a network of infected machines called a **botnet**.

### How the attack works

The target server is flooded with a huge volume of dummy traffic or malformed requests (for example **SYN floods** that leave half-open connections, or **HTTP floods** that hammer a web page). The server exhausts its **RAM, bandwidth, or CPU** and can no longer respond to real users.

<DosVsDdos />

### DoS vs DDoS

A plain **DoS** comes from a **single source/IP**, so it is relatively easy to identify and block. A **DDoS** comes from **thousands of distributed machines worldwide**, so blocking one address does nothing — which makes it far harder to filter and far more powerful.

<Callout type="danger">
**Example:** a hacktivist group uses a botnet of millions of infected IoT devices (smart cameras, routers) to flood an e-commerce site with billions of requests on Black Friday, crashing it and halting millions in sales.
</Callout>

**Defences:** firewalls and rate limiting, traffic filtering, content-delivery networks (CDNs), and specialised anti-DDoS / scrubbing services.

## SQL injection (SQLi)

**SQL injection is a code-injection attack in which the attacker inserts malicious SQL statements into a website's input fields, tricking the backend database into running unintended commands.** SQL is the language used to query databases (to fetch usernames, passwords, records); if user input is not checked, that input can be executed as **code** instead of treated as data.

### How it works

If a website builds its database query by directly pasting in whatever the user typed, an attacker can type SQL commands instead of a normal value. The database cannot tell the difference and runs the injected code as a legitimate instruction.

<SqlInjection />

### Impact

- Bypassing login/authentication.
- Reading or altering confidential records.
- **Dumping the entire database**, including passwords and financial data.
- Deleting data or, in severe cases, taking over the server.

<Callout type="tip">
**Classic example:** entering `' OR '1'='1` in a login field. Because `'1'='1'` is always true, the WHERE condition always passes, so the database authenticates the attacker **without a valid password**.
</Callout>

**Prevention:** **parameterized queries / prepared statements** (the key defence), strict **input validation**, escaping special characters, least-privilege database accounts, and web application firewalls.

## Buffer overflow

**A buffer overflow occurs when a program writes more data into a fixed-size block of memory (a buffer) than that buffer was allocated to hold.** The extra data spills into the **adjacent memory**, corrupting or overwriting legitimate values — including, dangerously, the program's control information.

### How it works and how it is exploited

In memory-unsafe languages like **C and C++**, if the program does not check the size of input, an attacker can deliberately send input that is far too large. The overflow can overwrite the **return address / instruction pointer** that tells the CPU what to run next. The attacker carefully fills the overflow with malicious machine code called **shellcode**, so the program "jumps" to and executes the attacker's code — with the program's own privileges (often administrator/root).

<BufferOverflow />

### Why it is dangerous

- Program crash and data corruption (loss of **Availability/Integrity**).
- **Remote code execution** — the attacker runs arbitrary commands and can take full control.

**Prevention:** secure coding with strict **bounds/input-size checking**, using memory-safe languages or safe library functions, compiler protections (stack canaries, ASLR, DEP/NX), and prompt patching.
